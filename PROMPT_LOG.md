## Задание Средняя сложность №5: Создать docker-compose.yml, поднимающий Python, Go и Rust сервисы.
**Инструмент:** Claude Sonnet 4.6 (claude.ai)
### Промпт 1
**Промпт:** У меня есть три сервиса, которые должны работать вместе в Docker Compose. На основе приложений, загруженных в чат (Go-сервис для студентов, Rust-сервис лояльности, Python-шлюз), создай Dockerfile для каждого и общий docker-compose.yml, соблюдая следующие требования:
Общие требования
- Все сервисы должны собираться с помощью `docker compose up --build`.
- Используй многоступенчатую сборку (multi-stage) для Go и Rust, для Python – одностадийную.
- Для каждого сервиса определи healthcheck (где возможно).
- Используй переменные окружения для настройки взаимодействия.
- Для Go-сервиса: сборка с CGO_ENABLED=1 (так как используется SQLite), запуск в alpine:3.19 (или debian:bookworm-slim). Установить sqlite-libs, curl для healthcheck.
- Для Rust-сервиса: сборка в `rust:latest` (или `rust:1.90-slim`), запуск в `debian:bookworm-slim` с установленным `curl` (для healthcheck).
- Для Python-сервиса: базовый образ `python:3.12-slim`, установка зависимостей из `requirements.txt`, запуск через `uvicorn`.
Создать Makefile с командами:
- `make up` – `docker compose up --build`
- `make down` – `docker compose down -v`
- `make test` – запуск тестов (если есть) для Python, Go, Rust.
**Результат:** Выведено 2 верных файла в соответствии с требованиями
### Промпт 2
**Промпт:** Проведи ревью всех файлов.
**Результат:** Healthcheck для Rust-сервиса падает (unhealthy). В образе debian:bookworm-slim отсутствует curl, который используется в проверке.
Решение: В Dockerfile для Rust добавлена установка curl в runner-образ. Также увеличен start_period до 30 секунд. Обнаружен конфликт портов при запуске docker compose up
Результат: Порт 8080 на хосте уже занят (PID 4072, 24060).
Решение: В docker-compose.yml внешний порт Go-сервиса изменён с "8080:8080" на "8082:8080". Внутренний порт остался 8080, переменные окружения не менялись.
### Промпт 3
**Промпт:** Проведи повторное ревью всех файлов.
**Результат:** Go-тесты не проходят из-за CGO и драйвера go-sqlite3. Тесты требуют CGO_ENABLED=1 и наличия GCC. В среде Windows это вызывало ошибки.
Решение: Драйвер github.com/mattn/go-sqlite3 заменён на modernc.org/sqlite (чистый Go, не требует CGO). Исправлены импорты и имя драйвера с "sqlite3" на "sqlite". Тесты стали проходить
### Промпт 4
**Промпт:** Проведи финальное ревью всех файлов.
**Результат:** В проекте Python-сервис лежит в папке python, а не gateway.
Решение: В Makefile путь для test-python изменён на cd python && pip install -r requirements-dev.txt && pytest -v.
Все остальное корректно.
### Итого
- Количество промптов: 4
- Что пришлось исправлять вручную: созданы коммиты, проверка работы приложения без контейнера, код скопирован и перенесен из claude.ai. Собраны Docker-образы для всех трёх сервисов (Python, Go, Rust), проверка запуска контейнера.
- Время: ~3 часа

## Задание Средняя сложность №3: Настроить CI/CD, который собирает и пушит образы для всех трёх языков.
**Инструмент:** Claude Sonnet 4.6 (claude.ai)
### Промпт 1
**Промпт:** Настроить CI/CD, который собирает и пушит образы для всех трёх языков.
Q: Где хостится репозиторий?
A: GitHub
Q: Куда пушить образы?
A: Docker Hub
Q: Когда запускать пайплайн?
A: Push в main
**Результат:** создан файл .github/workflows/ci.yml в проекте py_go_rs_services. После этого GitHub Actions будет автоматически собирать и пушить образы при каждом пуше в ветку main (или по тегам)
### Промпт 2
**Промпт:** Выведи план работы дальше
**Результат:** 1)Создаёшь аккаунт на Docker Hub (если нет) и там генеришь токен — Account Settings → Security → New Access Token
2)Пушишь проект на GitHub
3)Заходишь в репозиторий → Settings → Secrets and variables → Actions → New repository secret — добавляешь DOCKERHUB_USERNAME и DOCKERHUB_TOKEN
4)Делаешь любой коммит в main — пайплайн запустится автоматически
